## 1. 版本目标

1. **阿里云 V1 不动核心逻辑**
2. **支持火山云最小成本接入**
3. **阿里云 + 火山云同时采集 + 同库存储**

---

## 2. 核心设计原则

| 原则 | 说明 |
|------|------|
| **不动旧代码** | 阿里云 V1 三个 Collector 内部逻辑完全不修改 |
| **最薄包装** | 一个 `AliyunCollector` 包装类，仅做委托 + 写入 cloudProvider |
| **无独立 Normalizer 层** | 转换逻辑内聚在 Collector 内 |
| **执行边界控制** | 单云 try/catch，单云故障不拖垮整体 |
| **统一入库模型** | 单表 + `cloud_provider` 列区分 |
| **允许局部失败** | 单云失败不影响其他云 |

## 4. CloudCollector 接口（简化版）

```java
package com.aimira.monitor.cloud;

public interface CloudCollector {

    /** 云厂商标识，如 "ALIYUN"、"VOLCENGINE" */
    String getCloudProvider();

    /** 是否启用该云厂商采集（可通过配置开关） */
    boolean isEnabled();

    /** 采集账户余额 */
    Optional<BalanceHistory> collectBalance();

    /** 采集费用信息 */
    Optional<BillingHistory> collectBilling();

    /** 采集云资源列表 */
    List<ResourceInfo> collectResources();
}
```

> 只有 4 个方法。不包含 `applyCloud()`、不包含 `buildGlobalId()`。cloudProvider 由各 Collector 内部直接写入。

## 5. Entity 层改动

三个 Entity 各加一个字段：

```java
// BalanceHistory.java — 加字段
@Column(length = 32, nullable = false)
@Builder.Default
private String cloudProvider = "ALIYUN";

// BillingHistory.java — 加字段（同上）

// ResourceInfo.java — 加字段（同上）
```

> `@Builder.Default` + 数据库列默认值 `'ALIYUN'`。是否需要迁移脚本视实际部署情况决定，见下方数据库小节。

### 数据库

`cloud_provider` 列通过以下方式之一加到生产库，**根据实际部署情况选择**：

#### 方案 A：Flyway / 手动 SQL（推荐 — 有存量数据时必选）

```sql
-- V2__add_cloud_provider.sql
ALTER TABLE balance_history ADD COLUMN IF NOT EXISTS cloud_provider VARCHAR(32) DEFAULT 'ALIYUN';
ALTER TABLE billing_history ADD COLUMN IF NOT EXISTS cloud_provider VARCHAR(32) DEFAULT 'ALIYUN';
ALTER TABLE resource_info   ADD COLUMN IF NOT EXISTS cloud_provider VARCHAR(32) DEFAULT 'ALIYUN';

-- 回填存量（如果 ALTER 时未设 DEFAULT 或存量行仍为 NULL）
UPDATE balance_history SET cloud_provider = 'ALIYUN' WHERE cloud_provider IS NULL;
UPDATE billing_history SET cloud_provider = 'ALIYUN' WHERE cloud_provider IS NULL;
UPDATE resource_info   SET cloud_provider = 'ALIYUN' WHERE cloud_provider IS NULL;
```

- ✅ `DEFAULT 'ALIYUN'` 保证存量行自动填充，`NOT NULL` 约束安全
- ✅ `IF NOT EXISTS` 幂等，重复执行不报错
- ✅ `ResourceInfo.saveOrUpdate` 首次采集不会因 NULL cloudProvider 产生重复数据
- ⚠️ 如果项目未引入 Flyway，直接手动执行上述 SQL 效果相同

#### 方案 B：仅依赖 `ddl-auto: update`（无存量数据 / 开发环境可用）

- Hibernate 自动加列，但**不保证**生成 `DEFAULT` 子句
- 如 Entity 设 `nullable = false` 且表中有存量行 → **PostgreSQL 会报错**（列无 DEFAULT 无法满足 NOT NULL）
- 如 Entity 设 `nullable = true` → 加列成功，但存量行 cloud_provider 为 NULL
- **开发环境 H2 `create-drop` 无此问题**，容易掩盖生产风险

#### 决策逻辑

| 场景 | 推荐方案 |
|------|---------|
| 生产库有存量数据 | 方案 A（Flyway 或手动 SQL） |
| 生产库全新部署（空库） | 方案 B 可行 |
| 开发 / 测试环境 | 方案 B（H2 create-drop） |
| `nullable = false` 必须生效 | 方案 A |

> `@Builder.Default "ALIYUN"` 保证应用层读取时不会空指针，但不解决数据库层 `NOT NULL` 约束与存量 NULL 的冲突。**建议有存量数据时走方案 A，这是最安全的路径。**

### AliyunCollector

```java
package com.aimira.monitor.cloud;

import com.aimira.monitor.collector.BalanceCollector;
import com.aimira.monitor.collector.BillingCollector;
import com.aimira.monitor.collector.ResourceCollector;

@Component
public class AliyunCollector implements CloudCollector {

    private final AliyunConfig aliyunConfig;
    private final BalanceCollector balanceCollector;
    private final BillingCollector billingCollector;
    private final ResourceCollector resourceCollector;

    @Override
    public String getCloudProvider() { return "ALIYUN"; }

    @Override
    public boolean isEnabled() { return aliyunConfig.isCollectorEnabled(); }

    @Override
    public Optional<BalanceHistory> collectBalance() {
        return balanceCollector.collect()
            .map(h -> { h.setCloudProvider("ALIYUN"); return h; });
    }

    @Override
    public Optional<BillingHistory> collectBilling() {
        return billingCollector.collect()
            .map(h -> { h.setCloudProvider("ALIYUN"); return h; });
    }

    @Override
    public List<ResourceInfo> collectResources() {
        return resourceCollector.collectAll().stream()
            .peek(r -> r.setCloudProvider("ALIYUN"))
            .toList();
    }
}
```

> 现有三个 Collector 内部一行不改。`AliyunCollector` 负责委托 + 写入 `cloudProvider`。

## 8. Scheduler 改造

```java
@Component
public class CollectScheduler {

    private final List<CloudCollector> collectors;
    private final BalanceService balanceService;
    private final BillingService billingService;
    private final ResourceService resourceService;

    @Scheduled(cron = "${scheduler.collect-cron:0 0 * * * *}")
    public void collectAll() {
        log.info("===== 多云采集开始, 共 {} 个云厂商 =====", collectors.size());

        for (CloudCollector c : collectors) {
            if (!c.isEnabled()) { continue; }
            collectFromCloud(c);
        }

        log.info("===== 多云采集完成 =====");
    }

    /**
     * 单云采集。只抓 Exception，不抓 Error（OOM 等应让 JVM 暴露而非静默）。
     * 单云失败不影响其他云。
     */
    private void collectFromCloud(CloudCollector collector) {
        try {
            log.info("===== [{}] 采集开始 =====", collector.getCloudProvider());
            long start = System.currentTimeMillis();

            collector.collectBalance().ifPresent(balanceService::save);
            collector.collectBilling().ifPresent(billingService::save);
            resourceService.saveOrUpdateBatch(collector.collectResources());

            long elapsed = System.currentTimeMillis() - start;
            log.info("===== [{}] 采集完成, 耗时: {}ms =====", collector.getCloudProvider(), elapsed);
        } catch (Exception e) {
            log.error("[{}] 采集失败（不影响其他云厂商）: {}",
                collector.getCloudProvider(), e.getMessage(), e);
        }
    }
}
```

| 边界类型 | 手段 | 说明 |
|------|------|------|
| **异常边界** | `catch (Exception)` | 单云 API/SDK/DB 异常不影响其他云 |
| **数据边界** | 每个 Collector 方法独立 | 单云失败不影响其他云已入库数据 |

> 不引入 `@Transactional` 自调用问题。事务由各 Service 方法自行管理（现状已如此）。

## 10. 告警 — 按云分别检测

### 余额告警

`AlarmDetector.detectBalanceAlarms()` 改为按 `cloudProvider` 分别获取最新余额：

```java
private List<AlarmRecord> detectBalanceAlarms() {
    List<AlarmRecord> records = new ArrayList<>();
    List<AlarmRule> rules = alarmRuleRepository.findByAlarmTypeAndEnabledTrue("BALANCE");

    List<String> providers = balanceService.getDistinctCloudProviders();
    for (String provider : providers) {
        BalanceHistory latest = balanceService.getLatestByCloudProvider(provider);
        if (latest == null) continue;

        for (AlarmRule rule : rules) {
            // 余额告警用 cloudProvider 做去重 key（替代 null），避免跨云冷却互相抑制
            if (isInCooldown(rule.getId(), provider)) continue;
            if (isTriggered(latest.getBalance(), rule.getThreshold(), rule.getOperator())) {
                String message = buildBalanceAlarmMessage(rule, latest, provider);
                AlarmRecord record = weComNotifier.send(rule.getId(), message);
                // resourceId 存 provider 以区分跨云冷却
                record.setResourceId(provider);
                alarmRecordRepository.save(record);
                records.add(record);
            }
        }
    }
    return records;
}
```

### 到期告警 — 无需改动

`getResourcesWithExpiry()` 拉取所有云资源，`(ruleId + resourceId)` 去重。跨云 resourceId 碰撞概率极低，v2 不做处理。

### 告警消息模板

```
## ⚠️ {provider} 余额告警
> 规则: **余额不足告警**
> 当前余额: **423.50 CNY**
> 告警阈值: **500.00 CNY**
> 检测时间: 2026-07-03 14:00:00
```

| 改动项 | 说明 |
|------|------|
| 余额告警 | 按 `cloudProvider` 分别检测，独立判断阈值 |
| 余额去重 | `(ruleId + cloudProvider)` 替代 `(ruleId + null)` |
| 到期告警 | 无需改动 |
| 消息模板 | 告警标题加 `{cloudProvider}` 标识 |

### DataQueryController — 查询接口同步改造

`/api/query/*` 三个接口同样加可选 `cloudProvider` 参数，与 Dashboard 模式一致：

```text
GET /api/query/resources?cloudProvider=VOLCENGINE                 ← 只看火山云资源
GET /api/query/resources                                          ← 全量（向后兼容）
GET /api/query/resources/stats?cloudProvider=ALIYUN                ← 按云统计
GET /api/query/balances?cloudProvider=VOLCENGINE&start=...&end=...
GET /api/query/billings?cloudProvider=ALIYUN&start=...&end=...
```

- `resources` 多条件搜索：`@RequestParam(required = false) String cloudProvider` → 追加 WHERE 条件
- `resources/stats`：`countByResourceType` / `countByRegion` 增加 `AND cloudProvider = :provider` 变体
- `balances` / `billings`：Repository 查询方法增加 `AndCloudProvider` 后缀变体

> 所有接口不传 `cloudProvider` 时保持全量查询，向后兼容。

> **2026-07-06 SDK 坐标修正**：需求文档原写的 `com.volcengine:volcengine-java-sdk:2.0.5` 不存在。火山引擎 Java SDK 采用**按服务拆模块**方式发布，实际使用以下 4 个独立 artifact：
>
> ```xml
> com.volcengine:volcengine-java-sdk-core:1.0.75       <!-- 核心库 -->
> com.volcengine:volcengine-java-sdk-billing:0.2.20     <!-- 余额 + 账单 -->
> com.volcengine:volcengine-java-sdk-ecs:1.0.75         <!-- ECS -->
> com.volcengine:volcengine-java-sdk-rdsmysqlv2:1.0.75  <!-- RDS MySQL v2（支持列表查询） -->
> ```
>
> RDS 使用 `rdsmysqlv2` 而非 `rdsmysql`，因为 v1 只有单实例查询 `DescribeDBInstance`，v2 才有列表查询 `DescribeDBInstances`（复数）。
