# 踩坑记录 — 阿里云 SDK 集成问题

记录 AIMIRA Monitor 项目在集成阿里云 Java SDK 时遇到的坑及解决方案。

---

## 坑1：BSSOpenApi 域名写错导致 DNS 解析失败

### 现象

```text
java.net.UnknownHostException: bssopenapi.aliyuncs.com: Name does not resolve
com.aliyuncs.exceptions.ClientException: SDK.ServerUnreachable : Server unreachable
```

定时任务每小时报错，余额和账单采集全部失败。

### 原因

代码中 BSSOpenApi 的域名写成了 **`bssopenapi.aliyuncs.com`**，但这个域名在公网 DNS 中并不存在（`nslookup` 返回 Non-existent domain）。

正确的 BSS（费用中心）API 域名是 **`business.aliyuncs.com`**。

| 域名 | DNS解析 | 说明 |
|------|---------|------|
| `bssopenapi.aliyuncs.com` | ❌ 不存在 | **错误域名，不要使用** |
| `business.aliyuncs.com` | ✅ 正常 | **正确的 BSS/Billing API 端点** |
| `ecs.aliyuncs.com` | ✅ 正常 | ECS API 端点（这个是对的） |

### 修复

```java
// ❌ 错误
request.setSysDomain("bssopenapi.aliyuncs.com");

// ✅ 正确
request.setSysDomain("business.aliyuncs.com");
```

### 涉及文件

- `BalanceCollector.java` — 采集账户余额
- `BillingCollector.java` — 采集账单费用

### 经验教训

阿里云不同产品的 API 端点命名没有统一规律：
- ECS 是 `ecs.aliyuncs.com`
- BSS（费用中心）是 `business.aliyuncs.com`（不是 `bss.aliyuncs.com` 也不是 `bssopenapi.aliyuncs.com`）
- RDS 是 `rds.aliyuncs.com`

**接入前务必查看阿里云官方 API 文档确认正确的 Endpoint。**

## 坑2：旧版 SDK 默认使用 HTTP 而非 HTTPS

### 现象

异常日志中打印的请求 URL 使用 `http://` 协议：

```text
connection http://business.aliyuncs.com/?SignatureVersion=1.0&Action=... failed
```

### 原因

`aliyun-java-sdk-core:4.6.4` 中使用 `CommonRequest` 时，如果不显式设置协议类型，**默认走 HTTP**。
阿里云 API 要求 HTTPS，HTTP 请求可能被拒绝或重定向。

### 修复

创建 `CommonRequest` 后必须显式指定 HTTPS：

```java
import com.aliyuncs.http.ProtocolType;

CommonRequest request = new CommonRequest();
request.setSysProtocol(ProtocolType.HTTPS);  // 必须加这一行
request.setSysMethod(MethodType.POST);
request.setSysDomain("business.aliyuncs.com");
// ...
```

### 涉及文件

全部 3 个 Collector：

- `BalanceCollector.java`
- `BillingCollector.java`
- `ResourceCollector.java`

### 经验教训

旧版阿里云 SDK（4.x）的默认配置不够安全，**所有 `CommonRequest` 创建后必须显式设置 HTTPS 协议**。新版 SDK（5.x / 使用 `aliyun-*-sdk` 产品包）已默认 HTTPS，不需要额外设置。

## 坑3：AccessKeyId 通过异常日志泄漏

### 现象

```text
log.error("余额采集 API 调用异常: {}", e.getMessage(), e);
```

当 API 调用失败时，阿里云 SDK 的 `ClientException.getMessage()` 会返回**包含完整请求 URL** 的错误信息，URL 的 QueryString 中带有明文的 `AccessKeyId` 和 `Signature`：

```text
SDK.ServerUnreachable : Server unreachable: connection http://xxx/?...
&AccessKeyId=LTAI5t71dwehfH6Y9ih9NRtS
&Signature=h0f1AQuJERVgsLm1DE%2BrBmEdWUo%3D
... failed
```

任何有日志访问权限的人都能直接拿到 AK。虽然 Signature 是一次性的，但 **AccessKeyId 长期有效**，泄漏后可配合 Secret 进行攻击。

### 修复

**不要打印 `ClientException` 的原始消息**，只打印错误码：

```java
// ❌ 危险：e.getMessage() 包含完整 URL + AK
log.error("API 调用异常: {}", e.getMessage(), e);

// ✅ 安全：只打印错误码，完整信息放 debug
catch (ClientException e) {
    log.error("余额采集 API 调用异常: errorCode={}", e.getErrCode());
    log.debug("余额采集完整异常", e);  // debug 级别，生产不输出
    return Optional.empty();
}
```

### 涉及文件

全部 3 个 Collector：

- `BalanceCollector.java`
- `BillingCollector.java`
- `ResourceCollector.java`

### 经验教训

1. **永远不要信任第三方 SDK 异常信息的敏感度**，SDK 可能会把完整请求 URL（含认证参数）放入异常消息
2. **异常日志分层**：ERROR 级别只记录错误码、业务上下文；完整堆栈和细节放 DEBUG 级别
3. **日志脱敏是必须的**：建议在项目中引入日志脱敏工具或 AOP 切面，自动过滤 `AccessKeyId`、`Signature`、`Secret` 等敏感字段
4. 即使 Signature 是一次性的，**AccessKeyId 也不应该出现在日志中**

## 坑4：企业内网 DNS 无法解析部分阿里云域名

### 现象

部分阿里云域名在内网 DNS 服务器下解析失败，但公网 DNS 正常。

### 原因

企业内网 DNS 服务器（如 `10.10.3.1`）可能：
- 缓存了过期的 DNS 记录
- 对某些不常见的子域名不做递归解析
- 配置了域名白名单，未在名单内的域名拒绝解析

本项目测试结果：

| 域名 | 内网 DNS 解析 |
|------|--------------|
| `aliyuncs.com` | ✅ 正常 |
| `ecs.aliyuncs.com` | ✅ 正常 |
| `business.aliyuncs.com` | ✅ 正常 |
| `bssopenapi.aliyuncs.com` | ❌ 失败（域名本身也不存在） |

### 应对措施

1. 先确认域名本身是否正确（DNS 公共解析验证：`nslookup <domain> 8.8.8.8`）
2. 如果域名正确但内网 DNS 无法解析，在 Docker 容器中指定公共 DNS：

```yaml
# docker-compose.yml
services:
  app:
    dns:
      - 8.8.8.8
      - 114.114.114.114
```

3. 或在 JVM 层面配置 DNS 缓存时间：

```bash
# 缩短 DNS 缓存（默认 -1 永久缓存）
JAVA_OPTS="-Dsun.net.inetaddr.ttl=60"
```

## 坑5：API 响应 Code 判断不统一 — 成功被误判为失败，告警静默失效

### 现象

```text
WARN  c.a.m.collector.BalanceCollector - 余额采集API返回失败: code=200, message=success
DEBUG c.a.monitor.alarm.AlarmDetector - 无余额数据，跳过余额告警检测
INFO  c.a.monitor.alarm.AlarmScheduler - 告警检测完成，本次发送 0 条告警
```

日志打印 `code=200, message=success` 却说"返回失败"，且告警永远不触发。

打印完整响应体后发现 API 实际上是成功的：

```json
{
  "Code": "200",
  "Message": "success",
  "Success": true,
  "Data": {
    "AvailableCashAmount": "10.00",
    "AvailableAmount": "10.00",
    "Currency": "CNY"
  }
}
```

### 原因

阿里云不同 API 的**成功 Code 不统一**，有的返回 `"Success"`，有的返回 `"200"`：

| API | 成功 Code |
|-----|----------|
| `QueryBill`（费用） | `"Success"` |
| `QueryAccountBalance`（余额） | `"200"` |

代码只判断了 `"Success"`，`QueryAccountBalance` 返回的 `"200"` 被当成失败处理。

**连锁反应：**

```
Code 判断只认 "Success" → "200" 被当成失败 → Optional.empty()
→ 余额数据不落库 → balanceService.getLatest() 返回 null
→ AlarmDetector 跳过余额告警检测 → 告警静默失效（0 条告警）
```

### 修复

```java
// ❌ 错误：只判断 "Success"
if ("Success".equalsIgnoreCase(code)) { ... }

// ✅ 正确：同时兼容 "200" 和 "Success"
if ("200".equals(code) || "Success".equalsIgnoreCase(code)) { ... }
```

### 涉及文件

- `BalanceCollector.java` — 余额采集（Code 判断不完整）
- `BillingCollector.java` — 费用采集（目前返回 "Success"，但域名和 HTTPS 也有问题，见坑1、坑2）

### 经验教训

1. **判断 API 成功前必须先看实际返回值是什么**，不要假设只有一种 Code
2. **加日志看完整响应**是调试此类问题最快的方式
3. **单体 bug 会产生连锁反应**：Collector 层的一个判断错误 → 数据层为空 → 告警层静默失效 → 用户以为一切正常
4. **代码中判断成功失败要写得"宽容"**：白名单模式比黑名单模式更安全（明确列出视为成功的 Code，剩下的才是失败）

## 坑6：SWAS 轻量应用服务器 API 配置三连坑（域名 + 缺参数 + 误加 Code 校验）

### 现象

#### 阶段一：域名错误

调用 SWAS `ListInstances` API 直接报错：

```text
com.aliyuncs.exceptions.ClientException: SDK.ServerUnreachable : Server unreachable
```

#### 阶段二：域名修复后仍查不到实例

修复域名后 API 不再报错，但日志始终显示：

```text
INFO  c.a.m.collector.ResourceCollector - 当前 Region 无轻量应用服务器实例
```

实际上广州地域（`cn-guangzhou`）确实有一台实例。

#### 阶段三：加了 Code 校验反而拦住成功响应

补充 `RegionId` 并参考 BSS 的做法加了 Code 校验后：

```text
WARN  c.a.m.collector.ResourceCollector - SWAS API 返回失败: code=, message=
```

`code` 和 `message` 都是空的——API 实际是成功的，但被自己加的校验逻辑拦住了。

### 原因

SWAS API 与 BSS/ECS/RDS 的响应格式完全不同，一步接一步踩坑：

#### 问题 1：域名错误

SWAS 和 ECS/RDS 不同，**不支持中心域名**，必须使用**区域化域名**：

```
swas.<region-id>.aliyuncs.com
```

例如广州地域：`swas.cn-guangzhou.aliyuncs.com`

| 产品 | 域名格式 | 说明 |
|------|---------|------|
| ECS | `ecs.aliyuncs.com` | 中心域名 ✅ |
| RDS | `rds.aliyuncs.com` | 中心域名 ✅ |
| BSS | `business.aliyuncs.com` | 中心域名 ✅ |
| **SWAS** | **`swas.<region>.aliyuncs.com`** | **区域域名，无中心端点** ❌ |

#### 问题 2：缺少必填参数 `RegionId`

ECS/RDS 用中心域名 + SDK profile 的 region 即可自动路由。但 SWAS `ListInstances` API **必须显式传入 `RegionId` 查询参数**，光域名指向区域不够，API 没收到 `RegionId` 就不会返回该地域的实例。

> 域名修了仍查不到实例的根因——API 调用成功但返回空列表，排查很隐蔽。

#### 问题 3：错误套用 BSS 的 Code 校验模式

参考 BSS 的做法（[[踩坑记录.md#坑5]]）给 SWAS 也加了 Code 校验，但这**不适用于 SWAS**。不同产品的 API 响应格式不同：

| API 产品 | 成功响应格式 | 有无 `Code` 字段 |
|----------|------------|:---:|
| BSS（QueryAccountBalance） | `{"Code":"200", "Data":{...}}` | ✅ 有 |
| BSS（QueryBill） | `{"Code":"Success", "Data":{...}}` | ✅ 有 |
| ECS（DescribeInstances） | `{"Instances":{"Instance":[...]}}` | ❌ 无，数据直接返回 |
| RDS（DescribeDBInstances） | `{"Items":{"DBInstance":[...]}}` | ❌ 无，数据直接返回 |
| **SWAS（ListInstances）** | **`{"Instances":[...], "TotalCount":N}`** | **❌ 无，数据直接返回** |

SWAS 和 ECS/RDS 一样，成功时直接返回数据，**根本不含 `Code`/`Message` 字段**。`root.path("Code").asText()` 拿到空字符串 `""`，既不是 `"200"` 也不是 `"Success"`，就被当成失败拦住。

失败由 SDK 层通过 `ClientException` 抛出，不需要应用层额外从响应体判断。

### 修复

```java
// ❌ 错误1：SWAS 没有中心域名
request.setSysDomain("swas.aliyuncs.com");
// ✅ 正确
request.setSysDomain("swas." + aliyunConfig.getRegion() + ".aliyuncs.com");

// ❌ 错误2：缺少 RegionId（SWAS 必填，不像 ECS/RDS 可省略）
// ✅ 正确
request.putQueryParameter("RegionId", aliyunConfig.getRegion());

// ❌ 错误3：照搬 BSS 的 Code 校验 — SWAS 响应体没有 Code 字段
String code = root.path("Code").asText();
if (!"200".equals(code) && !"Success".equalsIgnoreCase(code)) { ... }
// ✅ 正确：不加 Code 校验，和 ECS/RDS 保持一致，失败由 SDK 抛 ClientException
JsonNode instances = root.path("Instances");
// 直接判断 instances 即可
```

### 涉及文件

- `ResourceCollector.java` — `collectSwasInstances()` 方法

### 经验教训

1. **阿里云不同产品的 API 端点策略不同**：SWAS 必须用区域域名，不像 ECS/RDS 有中心域名
2. **域名正确 ≠ 参数完整**：区域域名 + `RegionId` 缺一不可
3. **不要跨产品套用校验逻辑**：BSS 响应有 `Code` 字段的校验模式，不能直接搬到 SWAS/ECS/RDS 上。判断前先看实际返回值——这也正是 [[踩坑记录.md#坑5]] 的核心教训
4. **SWAS/ECS/RDS 这类产品 API 失败由 SDK 抛 `ClientException`**，应用层只需 catch 即可，不需要从响应体判断 Code
5. **域名表**（以广州 `cn-guangzhou` 为例）：

| 产品 | 正确域名 |
|------|---------|
| ECS | `ecs.aliyuncs.com` |
| RDS | `rds.aliyuncs.com` |
| BSS | `business.aliyuncs.com` |
| SWAS | `swas.cn-guangzhou.aliyuncs.com` |

## 坑7：ECS/RDS/SWAS 采集只查单个 Region，且 ECS/RDS 缺 `RegionId` 参数

### 现象

配置 `region=cn-hangzhou` 时，只能查到杭州的资源，广州、深圳等其他地域的资源全部遗漏。即使将 region 改为广州，其他区域又会漏掉。

### 原因

三个资源采集方法都只查询**单个 Region**：

```java
IAcsClient client = clientFactory.createClient();  // 只用默认 region
```

而且 **ECS 和 RDS 缺少 `RegionId` 请求参数**（SWAS 修复后已补上）。虽然 ECS/RDS 用中心域名，但 `DescribeInstances` / `DescribeDBInstances` 的 `RegionId` 是必填项。旧代码没传也没报错，因为 SDK 可能在签名中带了 region，或者阿里云 API 有默认回退——但无论如何，只查了一个区域。

### 修复

改为**遍历多区域**采集：

**1. AliyunConfig 增加 `regions` 列表：**

```java
/** 需要采集的地域列表 */
private List<String> regions = Collections.emptyList();

public List<String> getEffectiveRegions() {
    if (regions != null && !regions.isEmpty()) {
        return regions.stream().filter(r -> !r.isBlank()).collect(Collectors.toList());
    }
    return Collections.singletonList(region);  // 回退到单 region
}
```

**2. application.yml 支持多区域配置：**

```yaml
aliyun:
  regions: ${ALIYUN_REGIONS:cn-guangzhou,cn-shenzhen,cn-hangzhou,cn-hongkong}
```

**3. 每个采集方法遍历 regions：**

```java
for (String region : aliyunConfig.getEffectiveRegions()) {
    try {
        IAcsClient client = clientFactory.createClient(region);
        // ... 设置 RegionId = region，设置对应域名 ...
        // 单个 region 异常不中断，continue 继续下一个
    } catch (ClientException e) {
        log.error("{} 采集 API 异常: region={}, errorCode={}", type, region, e.getErrCode());
    }
}
```

关键变更：
- `return resources` → `continue`（单个 region 失败不中断其他区域）
- ECS、RDS 补上缺失的 `RegionId` 参数
- 所有 resource 的 `.region()` 用循环变量而非 `aliyunConfig.getRegion()`
- 原始响应日志改 DEBUG 级别（避免多区域 INFO 刷屏）

### 涉及文件

- `AliyunConfig.java` — 新增 `regions` 字段 + `getEffectiveRegions()` 方法
- `application.yml` — 新增 `regions` 配置项
- `ResourceCollector.java` — ECS/SWAS/RDS 三个方法全部改为遍历多区域

### 经验教训

1. **配置中的 region 决定了你能看到多少资源**，默认只有 `cn-hangzhou`，其他区域全漏掉
2. **ECS 和 RDS 的 `RegionId` 参数也是必填的**，不传相当于让 API 自行猜测
3. **多区域遍历时异常处理要改**：单个 region 失败应该 `continue` 而不是 `return`，避免一个区域挂了影响全局

## 坑8：国际站地域 eu-west-1 导致 InvalidOperation.NotSupportedEndpoint

### 现象

手动触发数据采集时，日志中 ECS、RDS、SWAS 采集在遍历到 `eu-west-1`（伦敦）时报错：

```text
com.aliyuncs.exceptions.ClientException: InvalidOperation.NotSupportedEndpoint
    at com.aliyuncs.DefaultAcsClient.getCommonResponse(...)
    at com.aimira.monitor.collector.ResourceCollector.collectEcsInstances(...)
```

每次采集都会在 `eu-west-1` 这一步失败，虽然单个 region 异常被 catch 后不阻塞其他区域（见 [[踩坑记录.md#坑7]] 设计的 `continue`），但每次采集任务都会在日志中输出 ERROR。

### 原因

`eu-west-1`（伦敦）属于**阿里云国际站**地域。本项目使用中国站账号（RAM 子账号），`DefaultProfile.getProfile("eu-west-1", ...)` 创建的 SDK 客户端无法通过中国站 Endpoint 处理国际站地域的 API 请求：

```
阿里云账号体系：
├── 中国站 (aliyun.com)
│   ├── cn-hangzhou / cn-guangzhou / cn-shenzhen ... ✅ 正常
│   └── cn-hongkong                                ✅ 可用
└── 国际站 (alibabacloud.com)
    └── eu-west-1 (伦敦)                            ❌ 中国站账号不支持
```

资源采集代码中，`AliyunClientFactory.createClient("eu-west-1")` 用伦敦地域创建 `DefaultProfile`，但中国站账号的 Endpoint 无法路由到国际站地域，SDK 内部端点解析失败 → `InvalidOperation.NotSupportedEndpoint`。

### 修复

从默认 `regions` 配置中移除 `eu-west-1`：

```yaml
# ❌ 错误：混入国际站地域
regions: ${ALIYUN_REGIONS:cn-guangzhou,cn-shenzhen,cn-hangzhou,cn-hongkong,eu-west-1}

# ✅ 正确：只保留中国站 + 香港（香港在中国站账号下可用）
regions: ${ALIYUN_REGIONS:cn-guangzhou,cn-shenzhen,cn-hangzhou,cn-hongkong}
```

如果确实需要监控国际站资源，需要单独创建一个国际站子账号，并在代码中区分中国站/国际站客户端（使用不同的 SDK Endpoint）。

### 涉及文件

- `application.yml` — 修改 `regions` 默认值

### 经验教训

1. **中国站和国际站账号不互通**，不要混配地域到同一个账号下
2. **香港（`cn-hongkong`）虽然地理上在境外，但仍然属于中国站（aliyun.com），中国站账号可用**
3. 判断地域是否属于中国站：登录阿里云控制台，查看地域列表 — 能看到的地域才是当前账号能用的
4. 多区域遍历时要确保每个地域都在当前账号的可用范围内，国际站地域要排除或用独立 SDK Endpoint

## 总结：集成阿里云 SDK 检查清单

在接入阿里云 OpenAPI 时，逐项确认：

- [ ] **Endpoint 域名正确**：查阅阿里云官方 API 文档确认产品 Endpoint，不要猜
- [ ] **强制使用 HTTPS**：`request.setSysProtocol(ProtocolType.HTTPS)`
- [ ] **响应 Code 兼容判断**：同时判断 `"200"` 和 `"Success"`，不同 API 返回不同
