## ✅ 已明确

### 1. 数据库迁移策略

- **原状态**：需求声�"不需要 Flyway 脚本"，依赖 `ddl-auto: update` 自动加列
- **风险**：PostgreSQL + 存量数据时，`ALTER TABLE ADD COLUMN ... NOT NULL` 无 DEFAULT 会直接报错
- **决策**：需求已更新为方案A/B框架（见需求文档第5节）。生产有存量数据 → 走 Flyway 或手动 SQL；空库 → ddl-auto 可行
- **关联影响**：`ResourceInfo.saveOrUpdate` 首次采集重复数据问题，通过方案A的 `DEFAULT 'ALIYUN'` 一并消除

### 2. AliyunCollector 启用开关

- **决策**：方案A — 加 `aliyun.collector.enabled` 配置（默认 true），与火山云 `volcengine.enabled` 模式一致
- **落地**：`AliyunConfig.java` 新增 `collectorEnabled` 字段，`AliyunCollector.isEnabled()` 改为读配置，`application.yml` 和 `.env.example` 同步新增
- **位置**：需求文档第6节、第12节、第14节

### 3. DataQueryController 改造

- **决策**：方案A — `/api/query/*` 三个接口各加可选 `cloudProvider` 参数，与 Dashboard 模式一致
- **落地**：`resources` 多条件搜索 + `resources/stats` + `balances` + `billings` 四个端点全部支持 `@RequestParam(required = false) String cloudProvider`，不传时全量查询向后兼容
- **位置**：需求文档第11节（新增 DataQueryController 小节）、第14节

### 4. 告警冷却机制的语义混用

- **决策**：方案A — 接受现状
- **理由**：方案B 涉及 `AlarmRecord` Entity + DB 改动，与"v2 最小改动"原则冲突。需求已明确到期告警 v2 不做处理，告警消息模板中加的 `{provider}` 标识已经足够区分
- **注意**：`AlarmRecord.resourceId` 在余额告警中存云厂商名（如 `ALIYUN`），前端展示时需知晓此语义

### 5. CollectScheduler 执行顺序

- **决策**：采纳建议，加 `@Order` 注解，阿里云 `@Order(1)`，火山云 `@Order(2)`
- **理由**：顺序本身无所谓，但确定顺序利于日志可读和排查

### 6. 余额告警空数据边界

- **决策**：采纳建议，JPQL 加 `WHERE b.cloudProvider IS NOT NULL`
- **理由**：避免 `getDistinctCloudProviders()` 返回 `[null]` 导致后续查询行为未定义

### 7. Dashboard 全量汇总"最新一条"规则

- **决策**：接收现状，v2 不做处理
- **理由**：需求已明确此行为，v3 再优化多卡片或求和

### 18. cloudProvider 字段在 API 响应中缺失 ✅ 已修复

- **现象**：启动应用（H2 dev profile）后，`/api/query/resources` 和 `/api/query/balances` 返回的 JSON 中不包含 `cloudProvider` 字段

- **修复**：`mvn clean compile` 清理编译后，`javap` 确认 `getCloudProvider()` 已存在，集成测试证实序列化正常
- **状态**：✅ 已修复（2026-07-03）

## 🔴 代码审查发现（2026-07-03 `/code-review` high effort）

### 20. Stream.peek() 误用于生产数据修改

- **位置**：[AliyunCollector.java:70](src/main/java/com/aimira/monitor/cloud/AliyunCollector.java#L70) `collectResources()` 方法
- **问题**：`.peek(r -> r.setCloudProvider("ALIYUN"))` 使用 `peek()` 修改元素。`peek()` 的 Javadoc 明确声明仅用于调试，JVM 有权在非调试流中跳过该操作
- **修复**：改为 `.map(r -> { r.setCloudProvider("ALIYUN"); return r; })` 或提取 `this::setAliyunProvider` 方法引用，与 `collectBalance()`/`collectBilling()` 保持一致
- **状态**：✅ 已修复（2026-07-03）

### 21. 缺少 (resourceId, cloudProvider) 数据库唯一约束 ✅ 已修复

- **位置**：[ResourceInfo.java:15](src/main/java/com/aimira/monitor/entity/ResourceInfo.java#L15) `@Table` 注解
- **修复**：
  1. Entity 加 `@UniqueConstraint(columnNames = {"resourceId", "cloudProvider"})`
  2. `saveOrUpdate` 加 `catch (DataIntegrityViolationException)` 降级为 UPDATE，并发场景兜底
- **状态**：✅ 已修复（2026-07-06）

### 22. CollectScheduler 单云三操作共用一个 try/catch ✅ 已修复

- **位置**：[CollectScheduler.java:58-73](src/main/java/com/aimira/monitor/scheduler/CollectScheduler.java#L58-L73) `collectFromCloud` 方法
- **修复**：余额/费用/资源三个操作各自独立 try/catch，云内操作级隔离
- **状态**：✅ 已修复（2026-07-06）

### 23. isInCooldown 方法有隐藏副作用

- **位置**：[AlarmDetector.java:154-174](src/main/java/com/aimira/monitor/alarm/AlarmDetector.java#L154-L174) `isInCooldown` 方法
- **问题**：方法名暗示纯 boolean 查询，但实际执行了 `alarmRecordRepository.findLastSentTimes()` 额外 DB 查询和 `log.info()` 格式化输出。违反命令-查询分离原则
- **建议**：拆分为 `isInCooldown(ruleId, resourceId)` 返回 boolean + `logCooldownDetail(ruleId, resourceId)` 负责日志，各自独立调用
- **状态**：🟢 低优先级（不影响功能正确性，属于代码整洁度问题）

### 24. cloudProvider 使用魔术字符串而非枚举

## 2026-07-06 修订记录

### SDK 坐标修正

需求文档原写的 `com.volcengine:volcengine-java-sdk:2.0.5` 不存在。火山引擎 Java SDK 采用按服务模块拆分方式发布，实际使用 4 个独立 artifact：

| artifact | 版本 | 用途 |
|----------|------|------|
| `volcengine-java-sdk-core` | 1.0.75 | 核心库（ApiClient、签名） |
| `volcengine-java-sdk-billing` | 0.2.20 | 余额 + 账单 |
| `volcengine-java-sdk-ecs` | 1.0.75 | ECS |
| `volcengine-java-sdk-rdsmysqlv2` | 1.0.75 | RDS MySQL（v2 支持列表查询） |

RDS 选择 `rdsmysqlv2` 而非 `rdsmysql` 原因：v1 只有 `DescribeDBInstance`（单实例查询），v2 才有 `DescribeDBInstances`（列表查询）。

### API Action 确认来源

| 常量 | Action 名 | 文档来源 |
|------|----------|---------|
| `BALANCE` | `QueryBalanceAcct` | 费用中心 API > 资金账户 |
| `BILLING` | `ListBill` | 费用中心 API > 账单管理 |
| `ECS_DESCRIBE_INSTANCES` | `DescribeInstances` | ECS API 概览（Version 2020-04-01） |
| `RDS_DESCRIBE_INSTANCES` | `DescribeDBInstances` | RDS MySQL API 概览（Version 2022-01-01） |

### VolcengineCollector 实现

使用真实 SDK 类型替换占位内部类：
- `collectBalance()` → `BillingApi.queryBalanceAcct()`：取 `cashBalance + creditLimit` 作为总余额，`availableBalance` 作为可用金额
- `collectBilling()` → `BillingApi.listBill()`：按月汇总 `originalBillAmount`，日均 = 月总额 / 当月天数
- `collectResources()` → `EcsApi.describeInstances()` + `RdsMysqlV2Api.describeDBInstances()`：按地域遍历采集 ECS 和 RDS
